# 《低代码开发技术》课程报告

## 校园失物招领平台设计与实现

---

## 一、摘要

本文基于郑州工商学院《低代码开发技术》课程考核要求，设计并实现了一个校园失物招领平台。平台采用现代 Web 开发技术栈，以 Next.js 14 作为前端框架，Supabase 作为后端数据库和认证服务，通过低代码理念快速构建了完整的失物/寻物信息发布、检索、认领等核心功能。本文详细阐述了系统的需求分析、数据建模、页面设计、业务逻辑实现及部署过程，并对开发过程中的经验教训进行了总结。

**关键词**：低代码、失物招领、Next.js、Supabase、Web应用

---

## 二、引言

### 2.1 项目背景

在高校校园生活中，师生丢失物品或拾到物品的情况时有发生。传统的失物招领方式主要依靠线下公告栏、校园广播或微信群等渠道，存在信息分散、传播范围有限、匹配效率低等问题。据调查，目前多数高校缺乏统一、高效的数字化失物招领平台，导致大量失物无法及时归还失主。

因此，开发一个便捷、高效的校园失物招领平台具有重要的现实意义。本课题立足于郑州工商学院的实际需求，运用低代码开发理念，旨在为全校师生提供一个集中管理失物信息的在线平台。

### 2.2 项目目标

本项目的核心目标是利用现代 Web 开发技术，构建一个功能完善的校园失物招领平台，实现以下功能：
- 方便的失物/寻物信息发布
- 高效的物品分类与检索
- 标准化的认领流程
- 可靠的数据存储与用户认证
- 后台管理与审核机制

## 三、需求分析

### 3.1 功能需求

经过对校园失物招领场景的深入调研，系统需要满足以下功能需求：

#### 3.1.1 用户端功能

1. **信息发布功能**
   - 失物招领发布：拾到物品的用户可以登记物品详情，包括物品名称、描述、拾到地点、时间、图片和联系方式。
   - 寻物启事发布：丢失物品的用户可以发布寻物信息，描述物品特征、丢失地点和时间。

2. **信息浏览与检索**
   - 分类浏览：按物品类别（电子产品、证件卡包、钥匙等）筛选查看。
   - 关键词搜索：支持按物品名称、描述、地点进行模糊搜索。
   - 最新信息展示：首页展示最新发布的失物/招领信息。

3. **认领功能**
   - 用户对匹配的物品发起认领申请。
   - 提交认领理由和证明信息。
   - 发布者审核确认认领。

4. **用户中心**
   - 查看自己发布的所有信息及状态。
   - 查看自己的认领申请记录。
   - 删除待审核的发布信息。

#### 3.1.2 管理端功能

1. **内容审核**
   - 审核新发布的失物/招领信息，确保内容真实有效。
   - 批准或拒绝信息发布。

2. **认领管理**
   - 查看所有认领申请。
   - 审核认领请求，批准或拒绝。

#### 3.1.3 系统功能

1. **用户认证**
   - 支持邮箱注册与登录。
   - 基于 Supabase Auth 的安全认证机制。

2. **图片存储**
   - 支持上传物品图片。
   - 使用 Supabase Storage 进行安全存储。

### 3.2 非功能需求

1. **性能要求**：页面加载时间不超过 2 秒，数据库查询响应在 200ms 以内。
2. **安全要求**：用户数据加密存储，实施行级安全策略（RLS）。
3. **可用性**：界面简洁直观，操作流程清晰，适配移动端和桌面端。
4. **可维护性**：代码结构清晰，模块化设计，便于后续功能扩展。

## 四、系统设计

### 4.1 技术架构

本系统采用现代 Web 开发技术栈，整体架构分为三层：

**前端层**：Next.js 14 (App Router) + TypeScript + Tailwind CSS
- 采用 React 服务端组件和客户端组件相结合的架构
- Tailwind CSS 实现响应式 UI 设计
- Lucide React 提供图标支持

**后端层**：Supabase
- PostgreSQL 关系型数据库
- Supabase Auth 用户认证
- Supabase Storage 文件存储
- Row Level Security 数据安全策略

**部署层**：Vercel + GitHub
- Vercel 提供零配置部署
- GitHub 进行版本控制和协作

### 4.2 数据模型设计

系统核心数据模型包含三个表：

#### 4.2.1 物品分类表 (categories)

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGSERIAL | 主键 |
| name | TEXT | 分类名称（唯一） |
| created_at | TIMESTAMPTZ | 创建时间 |

#### 4.2.2 物品表 (items)

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGSERIAL | 主键 |
| type | TEXT | 类型：lost（寻物）/ found（招领） |
| title | TEXT | 物品名称 |
| description | TEXT | 详细描述 |
| category_id | BIGINT | 分类ID（外键） |
| location | TEXT | 地点 |
| event_date | DATE | 事件日期 |
| contact_name | TEXT | 联系人 |
| contact_phone | TEXT | 联系电话 |
| contact_wechat | TEXT | 微信号 |
| image_url | TEXT | 图片URL |
| status | TEXT | 状态：pending/approved/resolved/closed |
| user_id | UUID | 发布者ID（外键） |
| created_at | TIMESTAMPTZ | 创建时间 |
| updated_at | TIMESTAMPTZ | 更新时间 |

#### 4.2.3 认领申请表 (claims)

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGSERIAL | 主键 |
| item_id | BIGINT | 物品ID（外键） |
| claimant_name | TEXT | 认领人姓名 |
| claimant_phone | TEXT | 联系电话 |
| claimant_wechat | TEXT | 微信号 |
| claim_reason | TEXT | 认领理由 |
| proof_description | TEXT | 证明信息 |
| status | TEXT | 状态：pending/approved/rejected |
| user_id | UUID | 认领人ID（外键） |
| created_at | TIMESTAMPTZ | 创建时间 |
| updated_at | TIMESTAMPTZ | 更新时间 |

### 4.3 安全策略设计

系统采用 Supabase 的 Row Level Security (RLS) 实现细粒度的数据访问控制：

1. **物品数据安全**
   - 所有人可查看已审核通过的物品
   - 用户只能查看和管理自己的物品（包括待审核状态）
   - 用户只能删除自己待审核状态下的物品

2. **认领数据安全**
   - 用户只能查看自己的认领申请
   - 物品发布者可以查看对自己物品的认领申请
   - 物品发布者可以审核认领申请状态

3. **图片存储安全**
   - 图片存储桶公开可读
   - 仅认证用户可以上传图片

### 4.4 页面路由设计

| 路由 | 页面 | 说明 |
|------|------|------|
| `/` | 首页 | 展示最新信息和统计数据 |
| `/lost` | 寻物列表 | 浏览所有寻物信息 |
| `/found` | 招领列表 | 浏览所有招领信息 |
| `/post` | 发布信息 | 发布失物或招领信息 |
| `/item/[id]` | 物品详情 | 查看详情，发起认领 |
| `/dashboard` | 用户中心 | 管理发布和认领记录 |
| `/admin` | 管理后台 | 审核内容和管理认领 |
| `/auth/login` | 登录 | 用户登录 |
| `/auth/register` | 注册 | 用户注册 |

## 五、功能实现

### 5.1 核心功能实现

#### 5.1.1 信息发布模块

信息发布页面支持两种类型的发布：寻物启事和失物招领。用户可以选择信息类型，填写物品名称、详细描述、分类、地点、日期和联系方式等信息。支持上传物品图片，图片将自动上传到 Supabase Storage 并生成公开访问链接。

发布流程：用户填写表单 → 验证输入 → 上传图片（如有）→ 将数据插入 items 表 → 跳转到对应列表页面。

#### 5.1.2 信息列表与检索模块

所有已审核通过的物品信息在列表中公开展示。用户可以通过以下方式检索：
- 分类筛选：按物品类别快速筛选
- 关键词搜索：基于标题、描述和地点的模糊搜索
- 类型区分：分别浏览寻物和招领信息

#### 5.1.3 认领模块

当用户找到匹配的失物或招领信息时，可以在物品详情页发起认领申请。认领流程包括：
1. 用户填写认领申请表（姓名、联系方式、认领理由、证明信息）
2. 系统创建认领记录，状态为"待处理"
3. 物品发布者在管理后台或用户中心查看认领请求
4. 发布者审核并决定批准或拒绝

#### 5.1.4 管理审核模块

管理员后台提供以下管理功能：
- 待审核物品列表：审核新发布的物品信息
- 全部物品管理：查看所有物品及其状态
- 认领申请管理：处理用户的认领请求
- 数据统计：展示平台运营数据概况

### 5.2 关键技术实现

#### 5.2.1 Supabase 集成

系统的数据层完全基于 Supabase 实现，通过 `@supabase/ssr` 包在服务端和客户端进行安全的数据库操作。

```typescript
// 客户端 Supabase 客户端
export function createClient() {
  return createBrowserClient<Database>(
    process.env.NEXT_PUBLIC_SUPABASE_URL!,
    process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY!
  );
}
```

#### 5.2.2 用户认证

使用 Supabase Auth 实现邮箱密码认证。通过 Next.js Middleware 自动管理会话状态，保护需要登录才能访问的页面。

#### 5.2.3 响应式设计

利用 Tailwind CSS 的响应式工具类，确保平台在不同屏幕尺寸下都有良好的用户体验。导航栏在移动端自动切换为汉堡菜单。

## 六、系统部署

### 6.1 部署架构

本系统采用前后端分离的部署架构：
- **前端**：部署在 Vercel 平台，享受自动 HTTPS、CDN 加速和零配置部署
- **后端**：使用 Supabase 云服务，提供数据库、认证、存储等一站式后盾服务
- **版本控制**：使用 GitHub 进行源代码管理

### 6.2 部署步骤

1. **Supabase 配置**
   - 在 Supabase 官网创建项目
   - 执行 SQL 脚本创建数据库表
   - 创建 Storage 存储桶
   - 配置认证设置

2. **代码推送**
   ```bash
   git add .
   git commit -m "初始提交：校园失物招领平台"
   git remote add origin https://github.com/用户名/仓库名.git
   git push -u origin main
   ```

3. **Vercel 部署**
   - 在 Vercel 导入 GitHub 仓库
   - 配置环境变量
   - 自动部署

## 七、问题与解决方案

### 7.1 开发过程中遇到的问题

1. **Supabase RLS 策略配置**
   - 问题：初始阶段未正确配置 RLS 策略，导致用户无法正确访问数据。
   - 解决方案：仔细设计每一条 RLS 策略，确保粒度和准确性。

2. **认证状态管理**
   - 问题：Next.js App Router 中服务端组件无法直接获取浏览器 Supabase 会话。
   - 解决方案：使用 `@supabase/ssr` 包的 Cookie 管理机制，通过中间件同步会话状态。

3. **图片上传优化**
   - 问题：图片上传后需要实时预览。
   - 解决方案：使用 FileReader API 在客户端生成数据 URL 实现预览。

### 7.2 经验总结

通过本次项目实践，我深刻认识到低代码开发方法论的核心价值：利用成熟的平台和服务（Supabase）快速搭建完整的后端基础设施，大幅减少了重复性工作，使开发团队能够更专注于业务逻辑实现。同时，React 组件化的开发模式提高了代码的复用性和可维护性，使项目结构清晰、易于扩展。

此外，在数据库设计和安全策略方面也收获了宝贵的经验。良好的数据模型设计是系统稳定运行的基础，而细粒度的权限控制则是保障数据安全的关键。

## 八、结论

本项目成功设计并实现了一个面向校园场景的失物招领平台。系统涵盖了信息发布、分类检索、认领管理、用户认证、管理后台等完整功能，满足了校园失物招领的基本需求。技术选型上采用了当前主流的 Web 开发技术栈（Next.js + Supabase + Vercel），既保证了开发效率，又确保了系统的性能和可扩展性。
